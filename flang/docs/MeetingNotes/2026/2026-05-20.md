* Design docs and/or RFCs  
  * [\[RFC\] Support for \-std=f2023](https://discourse.llvm.org/t/rfc-support-for-std-f2023/89608)  
    * There is a new proposal for how to proceed: [https://discourse.llvm.org/t/rfc-support-for-std-f2023/89608/24](https://discourse.llvm.org/t/rfc-support-for-std-f2023/89608/24)   
    * Related PR: [\#196410 Generalize \-pedantic to take an optional argument](https://github.com/llvm/llvm-project/pull/196410)   
      * Reviewers needed, as there are several different ways forward and detailed feedback is very much needed  
    * Related Issue: [https://github.com/llvm/llvm-project/issues/196409](https://github.com/llvm/llvm-project/issues/196409)   
    * Suggestion of getting \-std= working, with f2018 as the minimum standard (so no need to enforce standards prior to 2018\)  
    * Steve \- the way the compiler is written is not for 2018 semantics, it accepts everything from before that too.  Concern that if we say \-std=2018 that f03 and f08 will be expected too, and this could lead to mismatches in expectations from users.  If we’re going to do the \-std method, we should do it right so that the user experience meets user expectation.  He prefers the \-pedantic version.  
    * Idea, have \-std work as \-pedantic (after implementing the latter) and give an extra warning to say we are not strictly enforcing the standard  
    * Idea, have a separate flag that is specifically about the feature (i.e. SYSTEM\_CLOCK)  
      * Error out if using “bad” behavior of SYSTEM\_CLOCK, flag would be to allow those errors  
    * How to handle standards that don’t have SYSTEM\_CLOCK (ex. f77)?  
    * Potential for large increase in testing burden if we go the \-std route, due to the need to test for all differences between each new standard and all the ones we use before it (even if we cut it off at f2018 as the earliest standard)  
      * Current philosophy: If the old and new standards coexist happily, accept both, but if they clash then test for that specific feature behavior  
    * Steve \- this is only the second time we’ve run into this problem of conflicting standards for a given feature, if we start having a large number of situations like this, then it may make more sense to tackle this issue of enforcing strict standards compliance in a unified way  
    * Do we know if the \-std= flag is widely used by applications in the wild?  
      * Useful for teams developing and expecting portability across different compilers

* Issues of Note  
  * Unanswered Slack question about pow vs. powr: [https://app.slack.com/client/T5CSCA08L/C5C58TT32](https://app.slack.com/client/T5CSCA08L/C5C58TT32)   
    * In the "IEEE floating point" standard (also ISO/IEC 60559:2020-05), we read that there are two recommended exponentiation operations, identified as **pow(x,y)** and **powr(x,y)**. Lore has it that **pow** was modelled after C's pow function and **powr** was modelled after Fortran's \*\*. This compiler seems to implement \*\* (for real operands) with **pow** semantics. Other popular compilers also do so. My question is, was there any discussion of this choice and is there a record of it somewhere? Either way, I think it would be helpful to document this choice in the appropriate place.  
    * It may be too late to change the default semantics of R \*\* R to **powr**, but there is a decent chance that **powr** can be implemented to be faster than **pow** and some, at least, Fortran codes assume **powr** semantics and would rather not pay the price of **pow**.
