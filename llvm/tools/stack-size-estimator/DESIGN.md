# Stack Size Estimator Design Documentation

## Overview

The `stack_estimator.py` tool estimates the maximum stack usage of a program by analyzing its call graph and the stack frame size of individual functions. It leverages metadata generated by Clang's `-fexperimental-call-graph-section` and parsed by `llvm-readelf`.

The tool supports handling recursive calls (via Strongly Connected Components) and a mode to simulate a non-recursive environment (DAG mode).

## Core Concepts

*   **Call Graph (CG)**: A directed graph where nodes represent functions and edges represent function calls.
*   **Stack Usage**: The amount of stack memory (in bytes) reserved by a function's stack frame, extracted from the `.stack_sizes` section.
*   **Strongly Connected Component (SCC)**: A subgraph where every vertex is reachable from every other vertex. In a call graph, SCCs represent recursive cycles.
*   **Condensation Graph**: A Directed Acyclic Graph (DAG) derived by contracting each SCC into a single node.
*   **TypeID-based Indirect Call Resolution**: Indirect calls are resolved by matching the `TypeID` of a call site with the `TypeID` of potential target functions. This provides a conservative estimate of indirect call targets.

## Architecture

The tool operates in three main phases:

1.  **Input Parsing & Graph Construction**: Reads JSON input containing call graph info and stack sizes to build an in-memory graph. These JSON inputs are typically generated by `llvm-readelf --call-graph-info --stack-sizes --elf-output-style=JSON`.
2.  **Analysis**: Calculates the maximum cumulative stack usage starting from a specified entry point (e.g., `main`).
3.  **Reporting**: Outputs the results to the CLI or exports them to a structured JSON file for further analysis or visualization.

### 1. Input Parsing
*   **Input Format**: The tool expects a JSON object or a list of JSON objects. This allows for combining data from multiple translation units if they were processed separately.
*   **Graph Construction**:
    *   **Nodes**: Unique function addresses identified in the `CallGraph` section.
    *   **Direct Edges**: Established from `DirectCallees` metadata.
    *   **Indirect Edges**: Resolved conservatively by matching `IndirectTypeIDs` at call sites with the `TypeID` of potential target functions.
    *   **Stack Sizes**: Associated with each node by matching function names/addresses with the `.stack_sizes` section data.

### 2. Analysis Modes

The estimator supports two distinct modes of analysis:

#### A. Standard Mode (SCC-based)
This is the default mode, designed to handle recursion conservatively.

1.  **SCC Identification**: Uses Tarjan's algorithm to identify all Strongly Connected Components (SCCs).
2.  **Condensation Graph**: Builds a DAG where each node is an SCC.
3.  **Local Stack Contribution**: For each SCC, the tool calculates the **sum** of stack usages of all functions within it. This represents a worst-case scenario where all functions in a cycle might be on the stack simultaneously.
4.  **Path Calculation**: Performs a bottom-up traversal of the Condensation DAG.
    *   `MaxStack(SCC) = LocalStack(SCC) + Max(MaxStack(SuccessorSCCs))`
5.  **Result**: The max stack usage at the entry point's SCC.

#### B. Break Cycles Mode (DAG Mode)
This mode treats the call graph as a DAG by ignoring back-edges.

1.  **DFS Traversal**: Performs a Depth First Search starting from the entry point.
2.  **Cycle Breaking**: If an edge points to a node already in the current DFS path, it is ignored.
3.  **Memoization**: Caches the max cumulative stack for each visited node.
    *   `MaxStack(u) = LocalStack(u) + Max(MaxStack(v) for valid children v)`

### 3. Reporting

*   **CLI Output**: Prints the maximum stack usage, detected cycles, and the critical path (trace of functions) contributing to the maximum usage.
*   **JSON Export**: Generates a structured JSON file containing the reachable graph, calculated stack metrics, and identified cycles. This can be used by external visualization tools.
*   **Symbolization**: If `llvm-symbolizer` is provided, the tool resolves addresses to source locations (file:line) in the output.

## Data Structures

### Internal Graph
The `Graph` class (in `graph_helper.py`) uses an adjacency list representation.

### Analysis Results (`SccResults`)
A dictionary mapping an ID (SCC ID or Address) to its analysis data, including `max_stack_usage`, the list of `functions`, and the `next_scc_on_max_path`.

## Key Algorithms

*   **Tarjan's Algorithm**: For finding SCCs in $O(V+E)$.
*   **Topological Sort**: For processing the Condensation DAG.
*   **DFS with Memoization**: For the `break-cycles` calculation.

## Limitations

*   **Conservative Indirect Calls**: TypeID matching may include targets that are never actually called in practice, leading to overestimation.
*   **Recursive Sum**: Summing stack sizes in an SCC is a conservative upper bound.
*   **External Calls**: Calls to external libraries (where no call graph info is available) are treated as leaf nodes with 0 additional stack usage unless configured otherwise.

## Usage

```bash
# Basic Analysis
python3 -m llvm.tools.stack_size_estimator.stack_estimator <input.json> --entry-function main

# With Symbolization
python3 -m llvm.tools.stack_size_estimator.stack_estimator <input.json> --llvm-symbolizer-path /path/to/llvm-symbolizer

# Export for Visualization
python3 -m llvm.tools.stack_size_estimator.stack_estimator <input.json> --export-json report.json
```
